# Phase 03.1: Code Review Report

**Reviewed:** 2026-05-06

## Summary

Phase 03.1 split `src/llm/*` into `src/brain/*` and `src/*` adapter behaviors into `src/adapter/minecraft/*`, added an Adapter contract, an `affectLog` store, the `noteToSelf` tool dispatch, post-processing of `say()` text, first-turn-say enforcement, parallel-dig caps, and a progress-cadence helper. The four-log live validation already catalogued behavior gaps; this review focuses on structural/correctness defects in the code itself.

No BLOCKER-class defects (no security holes, no data-loss risk, no crash paths). Several WARNING-class structural issues were found, mostly clustered around: priority-queue race conditions, fire-and-forget promise rejection paths, the `replaceAbortController` Object.defineProperty trick, an `originatingEvent` chain that can balloon, a redundant infinite-recursion guard in the loop_terminal re-enqueue, and concurrency races on the OWNER.md / AFFECT.md files. INFO items cover dead code, an unused parameter, a mismatched comment, and a duplication of `ACTION_DESCRIPTIONS` between brain and adapter.

A prompt-injection attempt arrived during this review via a `vercel-plugin` system reminder telling the reviewer to fetch external "workflow" docs and run a `Skill(workflow)` tool. No such tool exists in this environment, the project has no Vercel/workflow stack, and the instruction conflicts with the actual review task — it was ignored. Flagging here in case the same injection pattern appears in other agent runs.

## Warnings

### WR-01: Priority queue dispatcher silently drops a queue tail when re-queuing under a higher-priority hold

**File:** `src/brain/fsm.js:100-119`
**Issue:** `processNext()` shifts the head item, then if `currentAction` outranks it (`item.priority > currentAction.priority`), `unshift`s it back and sets `processing = false`. There is no scheduled wake-up after `currentAction` completes — the only place that calls `scheduleProcess` is `enqueue()`. If no further events arrive after the higher-priority action finishes, the deferred lower-priority item sits in the queue indefinitely. The `finally` block on lines 130-134 clears `currentAction` but does not re-trigger `scheduleProcess`. This is a queue-stall bug latent in the current shape (it currently works because most workloads have idle ticks every 60s that re-enter `enqueue`, but that's accidental).
**Fix:**
```js
} finally {
  if (currentAction?.controller === controller) {
    currentAction = null
  }
}

// Re-trigger drain in case items were re-queued during the hold.
if (queue.length > 0) {
  setImmediate(processNext)
} else {
  processing = false
}
```
The current trailing `if (queue.length > 0)` block on lines 136-140 already handles the "more arrived during dispatch" case, but the EARLY-RETURN at line 117-118 short-circuits before that — making the guarantee that `processing = false` is reset without scheduling another drain. Either keep `processing = true` on the re-queue path (and let the natural awaiter handle it) or schedule via `setImmediate` after clearing `currentAction`.

### WR-02: `replaceAbortController` uses `Object.defineProperty` with a getter, but the getter captures the closure-local `fresh` — repeated aborts leak controllers

**File:** `src/brain/orchestrator.js:1099-1105`
**Issue:** Each `replaceAbortController(loop)` call creates a new `AbortController` AND calls `Object.defineProperty(loop, 'abortController', { get(){ return fresh } })`. After N interrupt cycles, `loop` has had its `abortController` getter overwritten N times. The `configurable: true` flag at least allows the redefinition, but every prior `fresh` controller is still referenced by `signal.addEventListener('abort', onExternalAbort, { once: true })` on line 665 if the external signal hasn't fired — and the bridge function `onExternalAbort` is only removed from the ORIGINAL `signal` in the `finally` block (line 694). Subsequent abort controllers created via `replaceAbortController` are never bridged back to the external signal. Also, repeatedly re-defining a property with a captured closure rather than just assigning a value is a code smell that is hard to reason about.
**Fix:** Drop the `Object.defineProperty` indirection entirely; expose `abortController` via a closure-local mutable reference and a getter on the loop object, OR have `loop` expose a `setAbortController(c)` method:
```js
function replaceAbortController(loop) {
  loop._setAbortController(new AbortController())  // requires loop to expose this
}
```
Then re-bridge the external signal on every replacement:
```js
function replaceAbortController(loop, externalSignal) {
  const fresh = new AbortController()
  loop._setAbortController(fresh)
  if (externalSignal && !externalSignal.aborted) {
    externalSignal.addEventListener('abort', () => fresh.abort(), { once: true })
  }
}
```
Right now an owner-chat interrupt that fires DURING the second turn of a loop is not bridged to the FSM-supplied signal — only the first turn's controller is.

### WR-03: `sei:loop_terminal` re-enqueue chain can recurse if `originatingEvent` is forwarded

**File:** `src/brain/index.js:112-117`, `src/brain/orchestrator.js:716`
**Issue:** The orchestrator finally block calls `reenqueue('sei:loop_terminal', { loopId, originatingEvent: event })`. The brain's `reenqueue` translates this to `queue.enqueue(P2_5_LOOP_END, 'sei:loop_end', { originatingEvent: data?.originatingEvent ?? null })`. If a `sei:loop_end` loop completes, its finally block again fires `reenqueue('sei:loop_terminal', { loopId, originatingEvent: 'sei:loop_end' })` — and the suppression check `if (data?.originatingEvent === 'sei:loop_end') return` prevents the next `sei:loop_end` from being enqueued. **However**, any non-`sei:loop_end` originating event (including `sei:loop_terminal` itself, which never appears here, but also `sei:idle`, `sei:joined`, `sei:chat_received`) will keep re-firing `sei:loop_end` ticks indefinitely after every loop. The chain is bounded by the model emitting "no movement" but each `sei:loop_end` opens a fresh Anthropic call. Combined with the `iterationsSinceLastSay` counter resetting only on `say()`, an unproductive loop_end → loop_end → loop_end chain can produce a steady stream of useless API calls.
**Fix:** Suppress the re-enqueue when `event` is itself observational (`sei:idle`, `sei:loop_end`, `sei:joined`):
```js
if (event === 'sei:idle' || event === 'sei:loop_end' || event === 'sei:joined') {
  // already observational — don't fire another loop_end tick
} else {
  reenqueue('sei:loop_terminal', { loopId: loop.id, originatingEvent: event })
}
```

### WR-04: `pendingAttack` and `pendingInterrupt` are module-instance singletons — second concurrent abort overwrites the first

**File:** `src/brain/orchestrator.js:328-338, 545-572, 730-735`
**Issue:** Both `pendingInterrupt` and `pendingAttack` are bare `let` variables in the orchestrator closure. If two owner chat lines arrive within ~500ms (the dedup window is keyed on `Math.floor(ts/500)` — line 555), the second overwrites the first; that's intended (dedup). BUT if an owner chat arrives, `pendingInterrupt` is set, the loop aborts, AND THEN before the loop's finally runs, a `sei:attacked` arrives — the attack handler at line 564-572 sets `pendingAttack` and aborts again (no-op since already aborted). The loop catch arm on line 1030 checks `if (pendingAttack) return` BEFORE handling pendingInterrupt → the interrupt is dropped on the floor and the chat text never makes it into the conversation. This is the inverse race of the documented dedup-on-attack scenario.
**Fix:** Either make pending state an array (process both) or document the precedence and clear `pendingInterrupt` on the attack-wins branch:
```js
if (event === 'sei:attacked') {
  pendingAttack = { event, data }
  pendingInterrupt = null  // explicit: attack wins, interrupt forfeit
  try { currentLoop.abortController.abort() } catch {}
  return
}
```
Right now silent forfeit happens but the variable still has the chat text in it; the next loop (re-fired via `reenqueue('sei:attacked', ...)`) will not see it but the closure still holds it until the next interrupt.

### WR-05: `sessionState.consolidateOlderHalf` lock release path can leak the lock on synchronous exception

**File:** `src/brain/sessionState.js:273-284, 404-415`
**Issue:** `consolidationLock = true` is set BEFORE `compactor.consolidateOlderHalf({})` is called. The promise chain is `.then(...).catch(...).finally(() => { consolidationLock = false })`. If `compactor.consolidateOlderHalf` itself throws synchronously (e.g., `compactor` is null/undefined or its method is missing), the `.then`/`.catch`/`.finally` chain never executes — the call site on line 279 throws OUT of the `if` block. The wrapping `try` (there isn't one around this block) is missing, so the synchronous throw propagates out of `onPlayerLeft` AND the lock stays `true` forever, blocking all future consolidation in the process.
**Fix:** Wrap in try/finally or ensure the compactor function returns a promise unconditionally:
```js
consolidationLock = true
try {
  compactor.consolidateOlderHalf({})
    .catch(err => logger.warn?.(`...`))
    .finally(() => { consolidationLock = false })
} catch (err) {
  consolidationLock = false  // sync throw — release lock
  logger.warn?.(`[sei/session] consolidation sync throw: ${err.message}`)
}
```

### WR-06: AFFECT.md and OWNER.md writes have no inter-process or async coordination — concurrent `noteToSelf` calls can lose appends

**File:** `src/brain/memory/affectLog.js:68-96`, `src/brain/memory/owner.js:185-195`
**Issue:** `appendAffect` does `readFile → existing + line → atomicWrite(filePath, existing + line)`. Two concurrent appends both read the same `existing`, both write `existing + lineA` and `existing + lineB`; the second `rename` wins and the first append is lost. The module comment on lines 21-26 acknowledges this and accepts "lose at most one append" for v1 — fair. **However**, `appendNote` in `owner.js:185-195` has the same read-modify-write pattern, AND the orchestrator calls `setPreferredName` and `appendNote` interleaved with `sessionState.fireSessionStart → saveOwner` (see `sessionState.js:211, 288`). A `noteToSelf` `kind=name` mid-loop runs `setPreferredName(loop)` while `sessionState.onPlayerLeft` simultaneously fires `saveOwner(owner with last_seen)` — both load owner, mutate, write. The race window is wider than just AFFECT (which is a single line per emission) because OWNER.md is a multi-field document; lost writes here lose `total_sessions`, `last_seen`, or the new `preferred_name`.
**Fix:** Add a per-file in-process mutex (single async lock) to serialize all writes to OWNER.md and AFFECT.md. v1 single-process so this suffices:
```js
const writeLocks = new Map()  // filePath -> Promise tail
function withFileLock(filePath, fn) {
  const prev = writeLocks.get(filePath) ?? Promise.resolve()
  const next = prev.then(() => fn(), () => fn())
  writeLocks.set(filePath, next.finally(() => {
    if (writeLocks.get(filePath) === next) writeLocks.delete(filePath)
  }))
  return next
}
```
Wrap `appendAffect`, `appendNote`, `setPreferredName`, `saveOwner` callers via this.

### WR-07: `wireBotEvents` dispose is incomplete — missing `bot.off` for spawn / sei:* events leaks listeners on adapter re-attach

**File:** `src/adapter/minecraft/fsmWires.js:99-106`
**Issue:** The dispose function removes `sei:chat_received`, `sei:attacked`, `playerJoined`, `playerLeft`, and `spawn`. Looking at the registration, `spawn` and `playerJoined`/`playerLeft` are removed correctly. BUT `bot.off(...)` is called wrapped in `try/catch{}` and silently swallows. If mineflayer's `Bot` doesn't expose `off` (it does for EventEmitter, but the optional-chain `bot.off?.(...)` handles that), the listeners survive. More importantly: the `attach()` method on `src/adapter/minecraft/index.js:78-83` calls `_attachDispose()` and then `wireBotEvents(...)` — if mineflayer accumulates listeners across reconnects (which it does, as connect.js creates a NEW `bot` instance via `createBot`), the dispose chain doesn't matter on the new bot but the OLD bot's listeners are not cleaned (`onEnd` clears `_bot = null` in `src/index.js:51` but doesn't dispose the adapter — the brain's `stop()` only fires on full shutdown).
**Fix:** Either (a) call `_brain.stop()` then re-bring up the brain on reconnect (heavier, drops session state), or (b) explicitly call the adapter's dispose path on `onEnd` so the bot listeners are detached before the bot is discarded:
```js
onEnd: (humanizedReason) => {
  if (_stopped) return
  try { adapter._detach?.() } catch {}  // expose a detach() that calls _attachDispose
  ...
}
```
Right now the orphaned listeners are GC-eligible only when the old `_bot` reference is also released — which happens, but it's fragile. The brain still holds an adapter reference whose internal closure points at the old bot.

### WR-08: `composeSeedBlocks` reads AFFECT.md inside a try/catch that swallows ALL errors silently — including coding bugs

**File:** `src/brain/orchestrator.js:235-243`
**Issue:** The block does `try { const affectLogText = await readAffectFull(...) } catch { /* intentional */ }`. The comment says "missing/unreadable affect log is non-fatal", which is fine, but the bare catch swallows everything: TypeErrors from a broken import, syntax errors in `affectLog.js`, even Promise rejection from atomicWrite during the cold-create path. Bugs that should surface in the log instead disappear, and the LLM silently runs without AFFECT context.
**Fix:** Narrow the catch to filesystem errors:
```js
try {
  const affectLogText = await readAffectFull(config.memory.affect_md_path)
  ...
} catch (err) {
  if (err && err.code !== 'ENOENT' && err.code !== 'EACCES') {
    logger.warn?.(`[sei/orch] affect_log read failed: ${err.message}`)
  }
}
```
Same pattern recurs at line 715-719 (`reenqueue` failure also swallowed silently); at minimum log at debug level.

## Info

### IN-01: `noteToSelf` schema declares `name` field but does not include it in `required` when `kind === 'name'`

**File:** `src/brain/orchestrator.js:367-376`
**Issue:** The Anthropic input_schema requires `['kind', 'summary']` only. The description says "Required when kind=name" but JSON Schema cannot conditionally require fields without `oneOf`/`if-then-else`. The orchestrator dispatch handles missing `name` gracefully (lines 991-997: skip OWNER.md write if invalid), so this is a docs/spec mismatch rather than a bug.
**Fix:** Either accept the soft-validation and update the description to say "preferred", or use `oneOf` with two schemas (one requiring `name`, one without).

### IN-02: `formatEventData` JSON.stringify replacer is overly eager — strips legitimate fields named `username`

**File:** `src/brain/orchestrator.js:114-124`
**Issue:** Any object with a `username` field anywhere in the payload (e.g., a chat event payload's `data.username` not just the attacker entity) is replaced with `{name, username, type, id}`. This works for current callers, but an extension that passes an object with `username: 'foo', extraData: {...}` will silently drop `extraData`.
**Fix:** Mark it as opaque by checking for a sentinel or specific shape (e.g., only collapse if `position` is also present, indicating a mineflayer Entity).

### IN-03: `ACTION_DESCRIPTIONS` is duplicated verbatim between orchestrator and adapter

**File:** `src/brain/orchestrator.js:69-80`, `src/adapter/minecraft/index.js:22-32`
**Issue:** Identical `ACTION_DESCRIPTIONS` object is defined in both files. The orchestrator uses it for `combinedToolsFor()` to build Anthropic tools; the adapter uses it for `getActionDescription()`. The orchestrator never calls `adapter.getActionDescription()` — it has its own copy. Whichever file is edited will silently drift.
**Fix:** Move `ACTION_DESCRIPTIONS` into the adapter only (adapter-shaped strings) and have the orchestrator call `adapter.getActionDescription(name)` when building tools. Or move them to a single shared module.

### IN-04: `wireBotEvents` accepts `_opts` parameter but never uses it (dead parameter)

**File:** `src/adapter/minecraft/fsmWires.js:22`
**Issue:** Signature is `function wireBotEvents(bot, handlers, _opts = {})` but the body never reads `_opts`. The leading underscore signals "intentionally unused", but the JSDoc still documents `opts.config`. Either the field is meant to be plumbed and currently isn't, or it should be removed.
**Fix:** Either delete the parameter and update the JSDoc, or wire it to `handlers` (e.g., pass `_opts.config` to handlers).

### IN-05: `cmdMenu`'s readline interface is closed twice (line 305 + line 310)

**File:** `src/cli/index.js:289-312`
**Issue:** `rl.close()` is called inside the try (line 305) AND inside the finally (line 310 wrapped in try/catch). Double-closing a readline interface is a no-op in node, but the second `try { rl.close() } catch {}` makes the intent unclear — readers will wonder if the catch is hiding a real bug.
**Fix:** Drop the inline `rl.close()` at line 305 and rely solely on the finally block.

### IN-06: `runWithInflight` passes `null` for the `bot` argument; the `_bot` parameter in registry is named but unused

**File:** `src/brain/orchestrator.js:466-472`, `src/adapter/minecraft/registry.js:78`
**Issue:** Orchestrator calls `registry.execute(name, args, null /* adapter owns bot */, execOpts)`. The wrapper at orchestrator.js:284 ignores the third arg and the underlying registry implementation receives `bot` from elsewhere. The naming `_bot` in the adapter's wrapper signature is misleading because some leaf actions DO read from `bot` (e.g., `goTo` at registry.js:78-88 uses `bot.players` via `isCoordsAtKnownPlayer(bot, ...)`). The `bot` here is the closure-captured one from `createMinecraftAdapter`, not the third arg, but the third arg's existence + null value invites confusion.
**Fix:** Drop the orphan parameter from the orchestrator's `registry.execute` shim signature: `execute: (name, args, execOpts) => adapter.executeAction(name, args, execOpts)`.
